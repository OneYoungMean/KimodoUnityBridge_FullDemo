# 常见问题与报错处理（QA）

这份文档收集了使用 Kimodo Unity Bridge 时常见的报错和卡点，按场景分组。遇到问题时，可以先用页面顶部的关键词在本文里搜一下报错里的字眼。

如果这里没有覆盖你的问题，请参照文末的"提交 Bug"一节把日志发给作者。

## 一、首次使用与环境相关

### 第一次生成卡很久 / 提示需要下载

第一次运行生成时，工具会自动下载模型并配置运行环境，体积大约 10G。这个过程是一次性的，请耐心等待，不是卡死。

面板上若出现类似 **"Model missing detected, update required, approximately N minutes"** 的提示，说明检测到模型缺失，正在准备下载，N 是预计分钟数。

### 第一次生成失败 / 中途报错

第一次生成有小概率失败，通常是网络波动导致下载没完成。**重新点一次生成**一般就能解决。

### 报错提到 "Directory does not exist" 或 "Create Kimodo Server"

运行环境还没初始化。前往 **Project Settings → Kimodo Server Manager**，点击 **Create Kimodo Server** 引导工具创建运行目录和服务器文件。

### 报错提到 "Failed to create runtime root from package template" / "Cannot resolve Kimodo package root"

工具找不到自己的包目录或模板文件，通常是安装不完整。建议重新通过 Package Manager 安装一次，或确认包目录没有被误删文件。

### 报错提到 "Downloaded archive did not contain the expected content"

下载的安装包内容不完整，多半是下载中断。可在 Server Manager 里点 **Try Fix（delete and reconfigure）** 触发自动修复，它会清掉损坏的部分并重新配置。

## 二、服务器与显存相关

### 报错 "server exit with code 1" 或服务器启动失败

服务器进程异常退出。这类问题作者需要日志才能定位，请把服务器日志发过来（路径见文末）。先尝试**重启 Unity** 后再生成，有时能直接恢复。

### 报错提到 "HF_HUB_OFFLINE" / "OfflineModeIsEnabled" / "Model prepare failed"

服务器正在准备或补下载模型，但当前 Hugging Face 环境处于离线模式，或本地模型目录不完整。常见日志形态是 `Model prepare failed`、`Cannot find an appropriate cached snapshot folder`、`HF_HUB_OFFLINE=0`。

如果你希望自动下载模型，请确认系统或 Unity 启动环境里没有把 `HF_HUB_OFFLINE`、`TRANSFORMERS_OFFLINE`、`HF_DATASETS_OFFLINE` 设为 `1`。然后停止服务器，删除项目目录下 `NvlabKimodoQuickServer~` 里的 `.run.lock`、`.bridge.pid`、`serverport`，并删除空的 `models/Kimodo-SOMA-RP-v1` 目录后重新启动服务器。

如果你必须离线使用，请先把完整模型和文本编码器放到 `NvlabKimodoQuickServer~/models` 下；空目录或只下载了一部分文件都会被判定为模型缺失。

### 报错提到 "Bridge process is already running"

已经有一个服务器进程在跑了。一般不影响使用；若状态卡住，重启 Unity 让它重新拉起即可。

### 报错提到 "Detected stale endpoint file" / "stale serverport record"

服务器进程其实没在运行，但残留了一个旧的端口记录文件。这是无害的提示，重启 Unity 或重新生成即可刷新。

如果同时存在 `.run.lock` 或 `.bridge.pid`，并且服务器面板反复显示旧端口或模型准备错误，可以先停止服务器，再删除这三个状态文件让工具重新拉起。

### 报错提到平台被禁用，如 "Bridge Windows/Linux platform disabled"

本地服务器支持 Windows、macOS 和 Linux 三个平台。这条报错说明当前平台在设置里被关掉了，或者运行平台全部被禁用（"All runtime platforms are disabled"）。检查 BridgeRuntimeSettings 里对应平台的开关是否打开。

### CUDA 平台偶尔走了 CPU 生成（变得很慢）

已知问题：CUDA 平台生成时有小概率会切到 CPU 管线，导致单次生成从几秒变成约一分钟。**重启 Unity** 一般能恢复到 CUDA。

### 显存不够 / 想降低显存占用

在片段或工具面板的 **VRAM Mode** 里选 **Low**，它使用量化过的编码器，约占 4G 显存（High 模式约 16G）。Kimodo 基础模型本身约占 2G。

## 三、生成与提示词相关

### 报错 "Prompt is empty."

提示词是空的。在 **Prompt** 框里用自然语言写一句你想要的动作再生成。

### 报错 "Editor is compiling or reloading scripts." / "compiling..."

编辑器正在编译或重载脚本，此时不能生成。等编译完成（右下角转圈消失）再点生成。

### 生成结果每次都一样

**Random** 没勾选时会用固定 Seed，结果自然一致。想要不同结果就勾上 **Random**，多生成几次挑一个满意的。

### 生成的动画播放卡顿

关键帧可能过密。打开片段 Inspector 的 **Animation Bake → Advanced**，启用 **Reduce Keyframes** 并适当调高误差容忍（Position / Rotation / Float Error）来精简冗余帧。

另外，若整体卡顿，可把 **Project Settings → Kimodo Server Manager** 里的 **Max Cached Clip** 设为 100。

### 遇到与 KimodoAnimationCache 相关的报错

前往 **Project Settings → Kimodo Server Manager**，点击 **Clear Clip Cache** 清理缓存即可解决。

## 四、角色与重定向相关

### 报错提到 "Custom Avatar is required" / "Custom Avatar must be a valid Humanoid Avatar"

当你关闭了 **Auto Retarget On Binding** 时，需要手动指定一个有效的 **Humanoid** Avatar。要么重新打开自动重定向，要么在 Custom Avatar 处指定一个正确的人形 Avatar。

### 报错提到 "...is null/invalid/non-humanoid"（avatar 相关）

工具拿到的 Avatar 为空、无效或不是人形。请确认：

- 角色模型在导入设置里 **Animation Type 设为 Humanoid**，并已正确配置 Avatar。
- 轨道确实绑定到了带 Animator 的角色对象上。

### 报错提到 "Animation track has no Animator binding" / "Binding ... is null"

动画轨道没有绑定 Animator。在 Timeline 里把这条 Animation Track 绑定到场景中的角色对象。

### 报错提到 "Scene animator avatar is not compatible with preview template"

场景角色的 Avatar 与预览模板不兼容，通常还是 Avatar 配置问题。按上面的 Humanoid 配置项检查。

### 报错提到 "Cannot resolve ModelImporter..." / "...after humanoid auto-setup import chain"

工具尝试从模型自动生成 Humanoid Avatar 时失败，多半是这个对象不是从模型文件实例化来的，或模型导入设置异常。请用一个标准的、可被识别为模型资产的角色重试。

## 五、Timeline 与片段相关

### 报错 "KimodoPlayableClip not found for command target." / "playable clip is null"

工具找不到目标片段，通常是片段在生成过程中被删除或选择丢失。重新选中片段再操作。

### 报错 "No selected timeline clip for constraint export."

没有选中可用于导出约束的片段。先在 Timeline 里选中目标 Kimodo 片段。

### 报错 "Inside mode requires CurrentClip."

InOut Constraint 选了 **Inside**，但当前片段无效。确认你操作的是一个正常的 Kimodo 片段。

### 报错提到 "Timeline inspected director is null"

当前没有正在编辑的 Timeline。请打开角色身上 PlayableDirector 引用的 Timeline 资产，进入 Timeline 编辑状态再操作。

## 六、Animator Tool 相关

### 报错 "Select AnimatorStateTransition or AnimatorState."

工具没有检测到选中对象。在 Animator 窗口里点中一个状态或一条过渡连线。

### 报错 "Outside mode requires AnimatorStateTransition selection."

InOut Constraint 选了 **Outside**，但你选中的是状态而不是过渡。Outside 模式需要选中一条过渡，或者改用其他约束模式。

### 报错 "No selected transition or state to apply." / "...apply context is invalid"

点 Apply 时选择已经丢失。重新选中状态或过渡，确认预览正常后再 Apply。

### 选中 BlendTree 后无法生成

目前暂不支持 BlendTree（混合树）类型。请选择普通的状态或过渡。

## 七、约束（Constraint）相关

### 报错 "override is disabled."（编辑窗口里）

编辑窗口需要 **useOverride** 处于开启状态。如果你在窗口打开时关闭了它，窗口就会失效。重新开启 useOverride 再编辑。

### 报错提到 "clip range not found" / "marker is null"

约束 Marker 找不到对应的片段范围，通常是 **Marker 被放到了所属片段的时间范围之外**。把 Marker 移回片段范围内即可。

### 编辑保存后没生效

已知问题：约束编辑保存偶尔有概率没写入成功。点击 **Refresh Cache**，或重新打开一次编辑窗口确认数据是否正确。

### 约束太多导致动作变形

约束放得越多、彼此越矛盾，生成越难同时满足，结果可能变形。建议从少量关键约束开始，不够再补。

## 八、提交 Bug

如果上面没有解决你的问题，欢迎提交日志帮助改进：

- **Unity 编辑器报错**：提交 Editor.log。
- **服务器报错**（例如 server exit with code 1）：把项目路径下 `NvlabKimodoQuickServer~/log` 目录里的内容发过来。Runtime 模式下日志在 StreamingAssets 目录下。

提交时尽量附上你做了什么操作、用的什么平台和显卡，方便更快定位。
